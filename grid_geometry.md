# グリッド幾何問題の考え方

グリッドの幾何問題で詰まりやすいのは、
「図としては分かるのに、何を固定してどう判定するかが曖昧」
という状態。

このタイプは、最初に実装へ行かず、情報を分解すると進みやすい。

---

## 1. 問題を自分の言葉で1行にする

例:

- 「全部の部分長方形について、180度回転で一致するものを数える」
- 「あるマスから上下左右に進める範囲で最短距離を求める」
- 「黒マス同士の距離や位置関係を調べる」

これをやる理由は、問題文の細かい日本語から一度離れて、
**何をする問題か** を自分の頭で再構成するため。

---

## 2. 求めるものを1件固定する

「何を数えるか」が分かったら、次に **答えの1単位** を固定する。

例:

- 部分長方形の個数を数える → 長方形 `(h1, h2, w1, w2)` を1個固定
- 到達可能マスを数える → マス `(i, j)` を1個固定
- 2点の関係を調べる → 2点 `(x1, y1), (x2, y2)` を固定

今回の ABC455 B は、問題文そのものが
`(h1, h2, w1, w2)` を答えの単位として与えている。
こういうときは、まずそれをそのままループ変数にする。

---

## 3. 条件を座標の式に直す

幾何問題は、図のイメージだけで止まると実装できない。
必ず `(i, j)` を使った式に落とす。

よく使う変換:

- 上: `(i - 1, j)`
- 下: `(i + 1, j)`
- 左: `(i, j - 1)`
- 右: `(i, j + 1)`
- 点対称: `(a + b - i, c + d - j)`
- 1次元化: `idx = i * W + j`

ABC455 B では、長方形 `(h1..h2, w1..w2)` における
`(i, j)` の点対称な相手は

```cpp
(h1 + h2 - i, w1 + w2 - j)
```

になる。ここが問題の本体。

---

## 4. 制約から計算量を逆算する

設計が立たないときほど、制約を見る。

今回:

```txt
H, W <= 10
```

この時点でかなり強く

- 全マスを見る
- 全長方形を見る
- 各長方形ごとに中身も全部見る

が許されると分かる。

長方形の個数は

```cpp
H * (H + 1) / 2 * W * (W + 1) / 2
```

最大でも `55 * 55 = 3025` 個しかない。
だから「まず全探索」を疑うべき問題だった。

---

## 5. 判定関数 `ok(...)` を先に考える

実装に入る前に、
「1件固定したとき、それが条件を満たすか」
を関数として想像する。

ABC455 B なら:

```cpp
bool ok(h1, h2, w1, w2)
```

の中で

```cpp
for (int i = h1; i <= h2; i++) {
    for (int j = w1; j <= w2; j++) {
        int ni = h1 + h2 - i;
        int nj = w1 + w2 - j;
        if (S[i][j] != S[ni][nj]) return false;
    }
}
return true;
```

と判定できる。

この形まで出れば、外側は「全探索して `ok` を呼ぶだけ」になる。

---

## 6. 問題文から実装に落とすテンプレ

詰まったら、コードを書く前にこの5行を書く。

```txt
・何を数える / 求めるか
・何を1個固定するか
・条件を座標の式で書くとどうなるか
・制約上どこまで全探索できるか
・1件の判定関数は作れるか
```

これで「何から書けばいいか分からない」状態をかなり減らせる。

---

## 7. よくある失敗

- いきなり各マスの周囲を見始める
- 問題文が与えている変数を使わず、別の見方で無理に整理する
- 幾何条件を図のまま持っていて、式にしていない
- 制約が小さいのに最適化を考えすぎる

特に競プロ初学者は、
**問題文に出てきた変数がそのままループ変数になる**
ケースを見落としやすい。
